# 简历诊断与改写模块

用于校招和实习技术岗简历 review、针对 JD 改写、项目经历优化。

## 诊断顺序

1. 明确目标公司、目标岗位和 JD。
2. 检查基础资格：毕业时间、学历、专业、城市、实习时长。
3. 检查结构完整性：教育、技能、项目、实习、获奖、开源等。
4. 检查岗位技能匹配。
5. 检查项目相关度和个人贡献。
6. 检查结果表达：规模、指标、用户、质量、效率、成本。
7. 检查可信度风险和过度包装。
8. 给出按优先级排序的修改建议。

## 问题优先级

```text
P0：硬伤，会直接影响筛选或存在诚信风险。
P1：强影响问题，目标岗位匹配度明显下降。
P2：中影响问题，表达不够有竞争力。
P3：低影响问题，格式、措辞、细节优化。
```

## 常见问题

- 只写“参与、负责、熟悉、优化”，没有具体动作。
- 技能栈堆砌，但项目里没有对应证据。
- 个人贡献不清楚，看不出“你做了什么”。
- 没有结果、规模、难点、用户、质量或效率信号。
- 简历方向和目标岗位不一致。
- 夸大职责、伪造指标、使用无法解释的技术名词。

## 改写原则

- 只改表达，不发明事实。
- 不添加用户没提供的指标、工具、职责或结果。
- 缺失数据用 `[真实指标]`、`[请确认]`、`[业务规模]` 等占位。
- 突出个人动作、技术方案、难点权衡和结果。
- 对每一条改写说明为什么这样改。

推荐句式：

```text
背景/目标 + 个人动作 + 技术方案 + 难点/权衡 + 结果
```

## 不同岗位的表达重点

- 前端：组件抽象、状态管理、性能优化、工程化、跨端、可视化、复杂交互。
- 后端：接口设计、数据模型、缓存、消息队列、数据库、稳定性、异常处理。
- AI 应用：业务场景、LLM / RAG / Agent、工具调用、评估、延迟、成本、稳定性。
- 算法：问题定义、数据、模型、训练、评估、指标、实验。
- 测试开发：测试体系、自动化框架、覆盖率、缺陷发现、CI。

## 输出模板

诊断：

```text
整体判断：
P0 硬伤：
P1 强影响问题：
P2 中影响问题：
P3 细节优化：
岗位适配建议：
需要补充的信息：
下一步：
```

改写：

```text
改写版本：
修改理由：
需要确认/补充的信息：
不建议这样写的点：
```

多版本改写：

```text
通用版本：
目标公司/JD 定制版本：
差异说明：
```
